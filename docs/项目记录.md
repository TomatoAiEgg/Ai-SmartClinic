# 项目记录

这份文档只记录 Ai-SmartClinic 当前仍然重要的架构基线、关键能力和后续判断标准。不再保留日常修补、局部命名调整、零散测试补丁之类的信息。

## 1. 当前系统基线

Ai-SmartClinic 当前是一个多服务智能挂号原型，主链路如下：

```text
frontend-miniapp
  -> gateway-chat-service
  -> supervisor-agent
  -> triage-agent / guide-agent / registration-agent
  -> patient-mcp-server / schedule-mcp-server / registration-mcp-server
  -> PostgreSQL / Redis
```

核心边界：

- AI 负责理解、路由、推荐和回复。
- 真正的数据写入由后端工作流和业务规则执行。
- `common-ai` 只提供模型基础设施，不承载业务 Prompt。
- MCP 服务只提供工具和数据访问，不直接生成聊天回复。

## 2. 已完成的关键能力

### 2.1 多 Agent 架构基线

- `supervisor-agent`：总控路由。
- `triage-agent`：症状到科室建议。
- `guide-agent`：导诊问答。
- `registration-agent`：挂号编排和确认流。
- `common-agent`：统一的 Agent 工作流抽象。

这套结构已经完成重构，不再处于“大拆大改”的阶段。

### 2.2 登录与鉴权

- 微信登录入口已接通。
- 网关负责登录、鉴权和 Token 管理。
- 登录态写入 Redis。
- `APP_AUTH_ENABLED=true` 作为默认企业级行为。

### 2.3 挂号主流程

`registration-agent` 已支持：

- `CREATE`
- `QUERY`
- `CANCEL`
- `RESCHEDULE`

统一规则：

- 写操作必须先返回预览。
- 用户确认后才真正执行写操作。
- 预览上下文由后端生成并校验，不能靠前端或模型自由拼接。

### 2.4 症状驱动挂号能力

当前已支持“先描述病情，再继续挂号”的智能链路：

- `supervisor-agent` 会把“病情描述 + 挂号诉求”优先路由到 `triage-agent`
- `triage-agent` 给出科室建议后，前端可继续进入挂号
- `registration-agent` 在缺少科室信息时，会引导先按症状分诊，而不是直接死板拒绝

这项能力是当前智能体体验的关键修正点。

### 2.5 Guide RAG

RAG 已从单个模块内的轻量检索升级为公共基建方向，当前具备：

- `common-rag` 公共模块
- pgvector topK 检索封装
- 统一检索状态和检索日志能力
- `knowledge_document / knowledge_chunk / knowledge_ingest_job / knowledge_retrieval_log` 统一 schema 草案
- triage、guide、registration-policy 先接入公共检索器，后续再逐步迁移历史知识表

### 2.6 MCP 数据层落地

以下 3 个 MCP 服务已经使用 PostgreSQL，而不是早期的纯内存模拟：

- `patient-mcp-server`
- `schedule-mcp-server`
- `registration-mcp-server`

## 3. 当前主要技术方案

AI 相关技术主线：

- `Spring AI`
- `Spring AI Alibaba`
- 模块内 Prompt / RAG / Rule / Workflow 组合编排
- 自定义 Agent 工作流抽象 `common-agent`

传统工程技术保持轻量说明即可：

- Spring Boot 3
- WebFlux
- MyBatis-Plus
- PostgreSQL
- Redis
- Nacos
- uni-app + Vue 3 + Pinia

## 4. 当前判断标准

如果后续有人再问“这个项目是否已经完成重构”，判断标准按下面这套：

- 多 Agent 模块边界已经稳定
- 关键工作流已经从大类堆积拆成可维护结构
- Guide 已接入轻量 RAG
- Registration 已形成规则驱动的确认式工作流
- 核心入口链路已有测试基线

所以当前结论是：

`架构重构主线已经完成，后续重点是联调、稳定性和真实环境落地。`

## 5. 当前验证基线

本轮重构完成后，已通过过的关键验证包括：

```powershell
mvn test -DskipITs
mvn -pl gateway-chat-service,supervisor-agent,triage-agent,guide-agent,registration-agent -am test
```

```powershell
mvn -Dtest=RegistrationFlowPolicyTest,RegistrationWorkflowServiceTest test
mvn -Dtest=RegistrationAgentServiceTest#shouldGuideSymptomOnlyCreateRequestsToTriageBeforeRegistration test
```

## 6. 仍然重要的后续事项

只保留真正还重要的：

1. 真实环境联调
前端小程序、微信登录、Redis、PostgreSQL、MCP 服务、DashScope 需要持续做整链路联调。

2. 路由能力继续升级
`supervisor-agent` 当前仍以规则路由为主，后续可以升级为结构化 LLM routing，但输出仍然只能是 route，不能直接输出业务结果。

3. 生产级能力补齐
包括幂等、审计、监控、限流、熔断、HIS 适配等。这些属于交付层工作，不再是本轮重构范围。
