## 3. 重点技术问题清单

### 3.1 RAG 整体流程

面试官问题：

- 说一下你对 RAG 流程的理解。
- RAG 检索不准、关键词命中差、上下文过长这些问题你怎么看？

你当时回答到的点：

- RAG 并没有“死”，只是传统向量检索有局限。
- 数据入库前需要处理 PDF、Markdown、Excel 等格式。
- 文档解析后需要清洗、切片、embedding、向量入库、检索、拼接上下文、调用大模型。
- 切片可以按 token、字段大小、标点符号处理。
- 切片可以做 overlap，例如 100-200 token 后下一段从 190-290 token 开始。
- 除向量检索外，还补充 BM25、知识图谱和 Agentic RAG。

更好的回答框架：

```text
文档接入
  -> 解析 PDF / Word / Markdown / Excel
  -> 清洗和结构化
  -> 按标题层级 / token / 语义切片
  -> metadata 标注
  -> embedding
  -> 向量库 / 搜索索引 / 图数据库
  -> query 改写
  -> 多路召回
  -> rerank
  -> 上下文压缩
  -> prompt 组装
  -> LLM 生成
  -> 引用溯源和结果评估
```

需要补强的点：

- 当时回答偏流程堆叠，缺少对“为什么这么设计”的判断。
- 可以补充 metadata、chunk 粒度、召回评估、rerank、引用溯源、答案可信度。
- 可以明确说明 RAG 不是万能检索，结构化数据应优先走数据库或搜索引擎。

### 3.2 PDF、图片和 OCR 处理

面试官问题：

- PDF 里通常图片和文字混杂，图片怎么处理？
- 普通 PDF、矢量 PDF、扫描 PDF 怎么区分？

你当时回答到的点：

- 矢量 PDF 可用 PDFBox 提取文本。
- 图片或扫描区域调用 OCR。
- OCR 慢且花钱，所以要提前封装工具，能直接解析就不走 OCR。
- 表格可能被拆碎，需要合并片段后再结构化存储。

更好的回答框架：

```text
先判断 PDF 类型：
  可提取文本 -> PDFBox / parser 直接抽取
  扫描图像 -> OCR
  混合页面 -> 文本层抽取 + 图片区域 OCR

再处理版面：
  标题 / 段落 / 页眉页脚过滤
  表格区域识别
  表格结构还原
  图片 OCR 文本合并
  metadata 保留页码、标题层级、坐标信息
```

面试关注点：

- 不希望听到“全部 OCR”这种粗暴方案。
- 更看重成本、速度、准确率和文档结构还原。

### 3.3 Excel / 财务表格是否适合做 RAG

面试官问题：

- 有表头的 Excel，下面全是公司每天收款、付款记录，怎么变成可检索？
- 这种数据直接做 RAG 合适吗？
- 数字本身没有关键词意义，怎么检索？

你当时回答过程：

- 一开始说可能向量化，甚至提到 RNN。
- 后续被追问后修正为：纯 RAG 不太合适，更适合 ES、MySQL、标题/科目标签、模糊查询。
- 表头和标准科目可以作为标签，明细数据根据 ID 再查出来处理。

更好的回答：

```text
财务 Excel 属于强结构化数据，不应该直接整体向量化。

推荐做法：
  1. 用 POI 解析 sheet、表头、字段类型、单位。
  2. 标准化字段，例如日期、科目、金额、收支类型、部门、项目。
  3. 明细入 MySQL / PostgreSQL / ClickHouse 等结构化库。
  4. 表头、科目、备注等文本字段可进 ES / OpenSearch。
  5. 用户自然语言问题先做意图识别和字段映射。
  6. 涉及统计、筛选、聚合的走 SQL。
  7. LLM 负责把自然语言转查询条件，以及解释查询结果。
```

该题暴露的问题：

- 对结构化数据、半结构化数据、非结构化文档的处理边界需要更清晰。
- 面试官希望你先判断数据类型，再选择检索方案，而不是默认“向量化”。

### 3.4 BM25、知识图谱和多路召回

面试官问题：

- BM25 和知识图谱有什么用？
- 图谱上挂了什么节点和关系？
- 这些能力是强制调用，还是交给大模型自己决定？

你当时回答到的点：

- BM25 用于增强关键词、缩写、专有词命中。
- 知识图谱用于增强实体关系。
- 项目里图谱关系偏简单，主要是文档引用关系。
- Agentic RAG 模式下模型可以判断是否调用不同召回方式；普通 RAG 下做成可配置链路。

更好的回答：

```text
BM25：
  适合精确词、编号、简称、专业术语、低频关键词。

向量召回：
  适合语义相近但字面不一致的问题。

图谱召回：
  适合实体关系、多跳关系、上下游链路、组织关系、产品关系。

多路召回融合：
  每路召回给出候选文档
  -> 合并去重
  -> 统一 rerank
  -> 根据 score、metadata、权限、时效过滤
  -> 进入上下文压缩和 prompt 组装
```

答题短板：

- 知识图谱的节点和边讲得不够具体。
- “文档引用关系”对业务图谱价值偏弱。
- 没有明确讲召回融合、重排和评估指标。

### 3.5 RAG 检索后如何判断结果是否真的相关

面试官问题：

- 相似度分数只是向量指标，不一定代表真正语义相关，你怎么进一步调整？

你当时回答：

- 粗排、精排、TopK。
- 用 BM25 和知识图谱再处理。

更好的回答：

```text
检索后不能直接把 TopK 全塞给模型，需要二次判断：

  1. rerank 模型重排。
  2. score threshold 过滤低相关结果。
  3. metadata 过滤，例如时间、部门、权限、文档类型。
  4. query-document 交叉编码判断。
  5. 多路召回结果投票或加权融合。
  6. 上下文压缩，只保留与问题直接相关片段。
  7. 对答案做引用溯源，避免模型脱离证据回答。
```

面试官关注点：

- 不只是“检索出来给 LLM 判断”。
- 企业级 RAG 要能控制错误上下文进入模型。

## 4. AI 编程工具相关问题

### 4.1 AI 工具如何理解大型代码库

面试官问题：

- Claude Code、Codex 这类工具如何处理和检索整个代码库？
- 代码行数很多，不可能全部塞进上下文，背后可能做了什么？

你当时回答：

- 类似 ES 的关键词 / 模糊检索。
- 可能先对项目做分层分析，知道每块干什么。
- 像树结构一样逐层定位，不是一锅端全部读进去。

更好的回答：

```text
AI 编程工具通常组合多种能力：

  1. 文件树和项目结构扫描。
  2. 关键词检索，例如 ripgrep。
  3. 语言服务 / AST / LSP 获取符号引用。
  4. embedding 或语义索引定位相关文件。
  5. 读取局部文件上下文。
  6. 根据错误日志、测试失败、调用链继续追踪。
  7. 小步修改后运行测试或静态检查闭环。
```

可补强点：

- 提到 AST、LSP、符号索引、调用链，比只说 ES 更工程化。
- 强调工具不是靠“理解整个项目”，而是靠检索、局部阅读和验证闭环。

### 4.2 AI 生成代码幻觉和自我修复

面试官问题：

- AI 编造不存在的方法、参数写错、代码有 bug，怎么让它恢复？
- 模型幻觉的成因是什么？

你当时回答：

- 用 `Claude.md`、`AGENTS.md`、skill 文档描述项目结构、接口和规范，减少幻觉。
- 让 AI 先分析需求、定位代码、给方案，再逐步实现。
- 遇到业务理解不清晰时，应该让 AI 和用户交互确认。
- 对幻觉成因提到了训练数据错误、数据不实时、推理过程、自注意力机制等，但底层原理不深。

更好的回答：

```text
减少代码幻觉：
  1. 给 AI 明确项目结构和模块边界。
  2. 要求先检索真实代码，再引用真实类名和方法名。
  3. 禁止凭记忆编 API。
  4. 小步修改，改完运行编译、测试、类型检查。
  5. 失败后基于错误日志重新定位。
  6. 复杂任务先写 plan/spec，再实现。

模型幻觉成因：
  1. 语言模型本质是概率生成，不天然保证事实正确。
  2. 上下文不足时会用训练分布补全。
  3. 检索信息不准确或缺失时会生成看似合理的内容。
  4. 长上下文下注意力分散，指令和事实约束变弱。
  5. 模型没有真实执行环境感知，必须靠工具调用和测试闭环纠偏。
```

答题短板：

- 底层原理只停留在概念层面。
- 如果面试官继续问 attention、Transformer、上下文窗口、工具调用机制，会比较被动。

### 4.3 上下文变长、任务跟踪和工作流

面试官问题：

- AI 有 Todo 或任务跟踪，但可能以为自己完成了，实际上没完成，怎么办？
- 有没有搭过稳健的工作流？

你当时回答：

- 上下文变多后模型能力会变弱，需要 review。
- 需要跟用户交互，自己也要跟着代码走。
- 可以用 spec-driven 规范和 workflow。
- 自己主要写过一些 skill 和 MD 文档，还没有搭过企业级工作流。

更好的回答：

```text
稳健 AI 工作流：
  1. 需求澄清：明确输入、输出、边界、验收标准。
  2. 影响分析：列出涉及模块、接口、数据和风险。
  3. 任务拆分：每个任务有完成定义。
  4. 实现前确认方案。
  5. 小步提交，避免大范围混改。
  6. 自动验证：编译、测试、lint、类型检查。
  7. 结果复核：检查 diff、日志、接口响应。
  8. 总结残留风险。
```

面试官关注点：

- 不满足于“让 AI 自己 review”。
- 更想听到明确的流程、验收标准、自动化验证和人为复核点。

### 4.4 AI Agent 权限控制

面试官问题：

- Agent 执行命令、改文件、甚至可能删库跑路，如何看待权限？
- 如果权限请求太多，人会麻木，如何避免误点放行？

你当时回答：

- 权限要分层，增删改查区别对待。
- 对危险命令做关键词匹配，例如 `rm -rf`。
- 让 AI 在请求授权时解释命令可能导致什么后果。
- 高风险权限要单独审批或更强交互。

更好的回答：

```text
AI Agent 权限体系：
  1. 默认沙箱运行。
  2. 读权限、写权限、执行权限、网络权限分离。
  3. 高危命令黑名单和语义风险识别。
  4. 对生产库、密钥、部署环境默认只读或禁止访问。
  5. 批量授权要限制命令前缀和作用目录。
  6. 授权弹窗显示风险摘要，而不是只显示原始命令。
  7. 关键操作要求二次确认或人工审批。
  8. 所有命令和文件修改留审计日志。
```

这个问题回答方向是对的，但可以更系统。

### 4.5 AI 改代码越改越坏怎么办

面试官问题：

- AI 改一个 bug 又冒出来一堆 bug，最后收拾不了怎么办？

你当时回答：

- 频繁使用 Git。
- 每完成一个功能就 commit。
- 出事后回滚到之前版本。
- 做备份和缓存。

更好的回答：

```text
防止 AI 越改越坏：
  1. 每次只改一个明确问题。
  2. 先写或补测试复现问题。
  3. 小范围 diff，避免跨模块大改。
  4. 每一步都运行最小验证。
  5. Git 分支隔离，必要时 revert 单个提交。
  6. 对不确定改动先做 spike，不直接混入主线。
  7. review diff，重点看边界条件和副作用。
```

## 5. Multi-Agent 和 AI 友好型文档

### 5.1 Multi-Agent

面试官问题：

- 你怎么看 AI Agent / Multi-Agent？
- 会不会人多手杂，多个 Agent 改同一个文件导致冲突？

你当时回答：

- Multi-Agent 是趋势，因为单 Agent 长上下文会衰减。
- 多 Agent 可以“专人做专事”，减少 memory 干扰。
- 要先由主 Agent 理解需求、拆任务、写详细开发文档。
- 子 Agent 按边界开发，最后由主 Agent review 和合并。

更好的回答：

```text
Multi-Agent 适合：
  需求拆解、代码审查、测试生成、文档整理、独立模块开发。

Multi-Agent 风险：
  上下文不一致、重复修改、接口契约冲突、合并冲突、责任不清。

控制方式：
  1. 主 Agent 统一拆任务和定义接口契约。
  2. 子 Agent 只能改自己负责的目录或文件。
  3. 每个任务有明确输入、输出和验收标准。
  4. 合并前做统一 review、测试和冲突检查。
```

### 5.2 AI 友好型需求文档

面试官问题：

- 什么叫 AI 友好型需求文档？
- 真的试过吗，会更好吗？

你当时回答：

- 普通自动生成的 init 文档很多废话，像 README，不一定适合 AI。
- 应该把真正有用的规范、模块职责、代码风格、接口说明写进去。
- 自己写过 debug skill 和工作流 skill，感觉有效。

更好的 AI 友好型文档应包含：

```text
项目目标
模块职责
目录结构
关键接口
核心业务规则
不能做的事情
代码风格
测试命令
运行命令
常见坑
验收标准
安全限制
```

这个点和你当前项目很契合，但是否转化成项目文档后续再决定。

## 6. 测试、CI 和部署

### 6.1 测试理解

面试官问题：

- 你对测试理解多少？
- 测试点写不完怎么办？
- AI 自己写功能又自己写测试，会不会既当运动员又当裁判？

你当时回答：

- 主要写过单元测试。
- 需要描述业务场景和边界。
- AI 写测试容易只覆盖空值、0 值等常见场景，并发和特殊边界不够。
- 可以用不同角色和独立上下文分别做开发、review、测试。

更好的回答：

```text
测试分层：
  单元测试：验证纯业务规则和边界条件。
  集成测试：验证数据库、Redis、外部服务、MCP/HTTP 调用。
  契约测试：验证跨服务 DTO 和接口协议。
  端到端测试：验证真实用户链路。
  回归测试：防止历史 bug 重现。
  性能 / 并发测试：验证热点资源、锁、幂等。

AI 写测试的问题：
  只会覆盖显而易见路径。
  容易根据当前错误实现写“迎合测试”。
  需要人定义验收标准和关键风险。
```

### 6.2 CI 和部署

面试官问题：

- 了解工业级测试流程吗？
- 有没有部署过生产环境？
- Docker / 云原生是否了解？

你当时回答：

- 知道提交合并请求后自动跑构建和单测的流程。
- 公司没有实际部署权限。
- 自己在学生服务器上部署过小 demo。
- Docker / 云原生了解不深。

短板很明确：

- CI/CD、Docker、生产部署、可观测性、回滚策略需要补。
- 这类岗位不一定要求深入 K8s，但至少要能讲清楚从代码到上线的基本链路。

推荐准备口径：

```text
代码提交
  -> CI 拉起
  -> 编译
  -> 单元测试
  -> 集成测试
  -> 静态检查
  -> 构建镜像
  -> 推送镜像仓库
  -> 部署测试环境
  -> 冒烟测试
  -> 灰度 / 发布
  -> 监控和日志观察
  -> 异常回滚
```

## 7. 场景题：仓储机器人拣货系统

这是整场面试最重要的系统设计题。面试官不是考机器人专业知识，而是看你能不能拆真实复杂业务。

### 7.1 题目背景

给你一个传统仓库和一个机器人：

- 机器人有轮子、机械臂、摄像头。
- 接收订单任务，任务包含商品名称、条码、数量、货架号。
- 需要把用户订单里的商品从货架取回出口，后续打包出库。

### 7.2 面试官逐步追问点

第一层：机器人怎么走到商品位置？

- 需要仓库地图。
- 需要货架编号和坐标。
- 需要知道机器人当前位置。
- 需要路径规划。
- 可用视觉识别货架标号辅助定位。

第二层：站到货架前后怎么拿商品？

- 需要知道货架层级、位置、商品摆放。
- 需要判断机械臂高度、抓取姿态。
- 不能只说“知道位置就拿”。

第三层：同一排商品叠放，抓哪个最安全？

- 需要判断接触面和支撑关系。
- 优先抓顶部、左右无遮挡、接触面少的商品。
- 如果目标被遮挡，需要先移开旁边商品，再恢复。

第四层：下一件商品怎么选？

- 不是简单拿完一个再随机拿下一个。
- 涉及路径规划、就近原则、订单合并、批量拣货。

第五层：如何确认没拿错？

- 标签、条码、视觉识别、二次扫描。
- 回到打包台后再校验订单和实物。

第六层：如何选择快递箱？

- 考虑体积、重量、形状、商品类型、易碎、保温、快递公司要求。
- 类似装箱问题，不只是“能不能放进去”。

第七层：包装材料影响体积怎么办？

- 包装后体积变化，需要重新估计。
- 可通过 3D 视觉或规则估算包装后外接长方体。

第八层：多个机器人同时运行有什么挑战？

- 路径冲突。
- 热点货架拥堵。
- 多订单合并。
- 资源调度。
- 中央调度可能过载。
- 分布式协商可能降低单点压力。

第九层：爆款商品导致货架拥堵怎么办？

- 热点资源限流。
- 排队、预约、缓冲池。
- 限制同一货架同时服务机器人数量。

第十层：事故、障碍物、机器人卡住怎么办？

- 机器人视觉发现障碍后上报。
- 中央系统广播禁行区域。
- 派维修或故障处理机器人。
- 机器人长时间不动时自检超时并上报。

### 7.3 你的答题表现

答得较好的点：

- 能跟着面试官逐层补充问题。
- 提到了地图、路径、货架编号、视觉识别、接触面、最短路径、标签校验、箱体容量、多机器人冲突、热点排队、中央调度和异常上报。
- 对真实系统会遇到“堵塞、误拿、体积变化、故障”的问题有基本意识。

明显不足：

- 开始时没有主动拆系统，等面试官一步步引导。
- 缺少标准系统设计结构，例如输入、状态、模块、流程、异常、扩展性。
- 对机器人细节不熟可以接受，但业务建模不够主动。
- “最短路径”“二分”“零一算法”等词出现得较随意，没有明确适用边界。
- 中央调度和分布式协同的取舍讲得不够清晰。

更好的答题框架：

```text
1. 任务输入
   订单号、商品条码、数量、货架位置、优先级。

2. 基础数据
   仓库地图、货架坐标、库存位置、机器人状态、通道状态。

3. 核心模块
   任务调度、路径规划、定位导航、视觉识别、机械臂抓取、复核校验、异常处理。

4. 单机器人流程
   接单 -> 规划路径 -> 到货架 -> 定位商品 -> 判断可抓取 -> 抓取 -> 校验 -> 放入周转箱 -> 下一件 -> 回打包台。

5. 多机器人调度
   任务分配、路径避让、热点货架限流、队列预约、冲突检测。

6. 异常场景
   找不到货、拿错货、商品掉落、货架拥堵、机器人卡住、通道阻塞、库存不一致。

7. 系统目标
   准确率、吞吐量、平均拣货时间、故障恢复时间、安全性。
```

## 8. 面试官最终反馈

面试官最后给出的核心反馈：

- 作为应届生，不会按多年经验工程师标准要求。
- 但真实 B 端业务会有很多细节，思考问题要更贴近真实场景。
- 复杂系统不是靠 AI 写代码就能交付。
- AI 时代仍然看重工程师在 AI 之外的掌控能力。
- 关键能力是：能不能写出可交付质量、能不能完成业务、能不能有效率地解决真实问题。

这句话是整场面试最重要的信息：

```text
我们都能用 AI 写代码，但是能不能写出可以交付的质量，能不能真的把业务完成，而且有一定效率，是另外一个故事。
```

## 9. 这场面试暴露的核心短板

### 9.1 技术广度够，但深度不足

有接触 RAG、BM25、知识图谱、Agentic RAG、AI 编程工具、Multi-Agent，但很多点停留在“知道概念”。

容易被追问打穿的点：

- Transformer / attention / 幻觉机制。
- RAG 评估和 rerank。
- 知识图谱实体和关系建模。
- 结构化数据为什么不适合直接向量化。
- 企业级测试、CI/CD、部署。

### 9.2 工程化表达不够系统

回答中经常出现“可能”“大概”“我感觉”，说明临场组织还不够稳。

建议改成：

```text
我的判断是...
原因有三点...
如果是生产系统，我会这样拆...
这个方案的风险是...
兜底方案是...
```

### 9.3 真实业务建模能力需要加强

场景题里面试官持续引导，说明他想看到你主动把复杂问题拆成模块、状态、流程、异常和指标。

后续需要练：

- 业务流程建模。
- 状态机。
- 资源约束。
- 并发冲突。
- 异常补偿。
- 审计和可观测性。

### 9.4 部署和测试偏弱

测试、CI、云原生、生产部署回答明显偏浅。

对全栈 / 后端 / AI 应用落地岗位来说，这块至少要补到能清楚讲：

- 单测、集成测试、契约测试、E2E 测试的区别。
- CI/CD 基本链路。
- Docker 镜像构建和运行。
- 环境变量和配置管理。
- 日志、监控、回滚。

## 10. 后续准备建议

### 10.1 RAG 项目准备

准备一个更成熟的项目话术：

```text
我的 RAG 项目不是只做向量检索，而是按数据类型分流：

非结构化文档：
  文档解析 -> 清洗 -> 切片 -> embedding -> 向量召回 -> rerank -> 生成。

结构化表格：
  POI 解析 -> 字段标准化 -> 入库 -> SQL/ES 查询 -> LLM 解释结果。

专有名词和编号：
  BM25 / ES 补强。

实体关系：
  图谱建模实体和关系，支持多跳查询。

最终统一做召回融合、重排、引用溯源和答案评估。
```

### 10.2 AI 编程工具准备

可以准备一套固定说法：

```text
我不会让 AI 直接大范围改代码。
我会先让它读项目规范和模块边界，再要求它定位真实代码。
复杂任务先写 spec，再分解任务。
实现时小步修改，改完跑测试。
高风险命令必须人工确认。
如果越改越坏，用 Git 小提交和测试结果回到上一个稳定点。
```

### 10.3 测试和部署准备

至少补齐：

- JUnit / Mockito / SpringBootTest。
- Testcontainers 或 mock 外部服务。
- Maven test / package。
- Dockerfile。
- docker compose 启动 Redis、MySQL、后端服务。
- GitHub Actions 或其他 CI 的基本流程。

### 10.4 场景题准备

以后遇到大场景题，先别急着给算法名，先用固定框架拆：

```text
1. 输入是什么？
2. 输出是什么？
3. 有哪些核心实体？
4. 核心流程怎么走？
5. 有哪些状态？
6. 有哪些资源约束？
7. 有哪些异常？
8. 如何保证一致性？
9. 如何监控和恢复？
10. 如何扩展到更大规模？
```

## 11. 这场面试对项目包装的启发

先不直接改项目，但后续如果要用当前项目服务面试，可以重点包装这些方向：

- 智能挂号不是简单聊天，而是 AI 理解 + 后端确认流 + 号源状态机。
- 结构化数据不走 RAG，患者、号源、挂号记录走后端服务和数据库。
- AI 只做意图识别、字段抽取、回复生成；写操作由后端规则控制。
- 挂号确认类似真实业务里的资源预约和交易确认。
- 号源是热点资源，后续可以讲锁号、释放、幂等、并发和审计。
- `common-ai` 和业务 Agent 分离，可以体现工程边界。
- 前端、网关、Supervisor、Agent、MCP 多模块链路可以体现全栈和系统拆解能力。

这些点等你确认后，再决定是否反向补进项目代码、README 或面试讲稿。
